# planning/

> **这里放什么**：路线图、定价策略、计划。
>
> **判断标准**：问自己 —— "这份文档是在回答：**我们要建什么 / 怎么规划？**"

---

## 典型内容

- **定价策略**：分层设计、支付通道配置、买断 vs 订阅的权衡
- **路线图 / 阶段计划**：按阶段 / Sprint 安排的功能计划
- **重构实施计划**：跨提交的大型变更计划（执行完后标"历史文档"）
- **产品策略细节**：比顶层 PRODUCT_PLAN 更深的计划

## 不属于这里

- ❌ PRODUCT_PLAN 本身 → **项目根**（和 README 并列 —— 它是项目门面级文档）
- ❌ 当前瞬时状态 → `brain/STATUS.md`（STATUS 是"现在在做"；planning 是"将要建"）
- ❌ 已执行完的部署 → `operations/`
- ❌ 历史决策理由 → `brain/DECISIONS.md`

## vs `brain/STATUS.md`

| 维度 | `brain/STATUS.md` | `planning/*.md` |
|---|---|---|
| 时间尺度 | 现在 / 本周 | 未来几周 / 几个月 |
| 更新频率 | 每次会话 | 偶尔 |
| 内容 | 具体到"下个要改的文件" | 抽象到"Q2 出支付闭环" |
| 长度 | 软上限约 80 行 | 无硬上限 |

**快速判断**：会话结束就被覆盖的内容进 STATUS；跨会话稳定的计划进 planning/。

## vs `DECISIONS.md`

- `planning/` = **还没执行**的计划、**将要做**的方向
- `brain/DECISIONS.md` = **已经定下来**的事 + 为什么这么定 + 为什么没选其他

一旦 planning 里的方向**定了**，对应的决策条目也要追加到 DECISIONS。

## 维护纪律

- 已执行完的计划 → 在顶部标"**历史文档**"，保留但不修改
- 计划调整 → 不要覆盖；在 DECISIONS 追加一条"方向调整"决策
- **当前生效的策略**（例如当前在用的定价模型 —— 不是计划，而是现行政策）可以放这里，标"当前生效"
