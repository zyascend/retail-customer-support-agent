# Phase 6.5: Security Architecture Audit & Documentation — Design Spec

**日期**: 2026-06-13
**状态**: 设计已确认，待实施
**来源**: docs/long-term-optimization-path.md Phase 6.5

## 目标

把项目最强的安全卖点变成可展示、可验证的证据：7-layer write guard、pending action、confirmation、no-write invariant、audit log 和 deterministic safety boundary。

Phase 6.5 是纯文档扩展 + 审计 + 截图，不写新代码。

## 设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 文档组织 | 扩展 `portfolio-architecture.md` 第 4 节 | 面试官单文档即可看到深度安全设计 |
| 覆盖率映射 | 内嵌大表（layer → block reason → test → eval） | 一目了然，无需切换文件 |
| 演示证据 | Workbench 截图 + 中文标注 | 展示实际运行 UI 证据，精准标注 |
| 实施范围 | 纯文档扩展，不补测试、不写工具 | 保持阶段边界，Phase 7 新增写操作时自然触发 guard 扩展 |

## Guard Block Reason 全覆盖清单（审计基线）

`WriteActionGuard.check()` 共产生 **28 个 distinct block reason**，分布在 8 个校验步骤（含 action type 预检）：

### Layer 0 — Action Type Validation

| # | Block Reason | 触发条件 |
|---|-------------|---------|
| 1 | `unsupported_in_mvp` | action 在 DEFERRED_WRITE_ACTIONS 中 |
| 2 | `unknown_write_action` | action 不在 WRITE_ACTIONS 中 |

### Layer 1 — Authentication

| # | Block Reason | 触发条件 |
|---|-------------|---------|
| 3 | `authentication_required` | `state.authenticated_user_id` 为空 |

### Layer 2 — Explicit Confirmation

| # | Block Reason | 触发条件 |
|---|-------------|---------|
| 4 | `explicit_confirmation_required` | `confirmed=False` |

### Layer 3 — Ownership Validation

| # | Block Reason | 触发条件 |
|---|-------------|---------|
| 5 | `ownership_violation` | order.user_id ≠ auth user；modify_user_address target ≠ self |
| 6 | `order_not_found` | order_id 在 DB 中不存在 |

### Layer 4 — Read-Before-Write

| # | Block Reason | 触发条件 |
|---|-------------|---------|
| 7 | `read_before_write_required` | order/user 未在 loaded_context 中 |

### Layer 5 — Policy Compliance

| # | Block Reason | 触发条件 |
|---|-------------|---------|
| 8 | `non_pending_order_cannot_be_cancelled` | cancel 非 pending 订单 |
| 9 | `invalid_cancel_reason` | cancel reason 不在允许列表中 |
| 10 | `non_pending_order_cannot_be_modified` | modify address/items/payment 非 pending |
| 11 | `non_delivered_order_cannot_be_returned` | return 非 delivered 订单 |
| 12 | `non_delivered_order_cannot_be_exchanged` | exchange 非 delivered 订单 |
| 13 | `exchange_item_count_mismatch` | exchange item_ids ≠ new_item_ids 数量 |
| 14 | `replacement_item_count_mismatch` | modify_items 数量不匹配 |
| 15 | `order_item_not_found` | item_id 不在订单中 |
| 16 | `replacement_item_not_found` | new_item_id 不在 DB variant 中 |
| 17 | `replacement_item_product_mismatch` | 新旧 item 不属于同一 product |
| 18 | `replacement_item_unavailable` | 新 item 的 available=False |
| 19 | `payment_method_not_owned` | 支付方式不属于用户 |
| 20 | `same_payment_method` | 新旧支付方式相同 |
| 21 | `gift_card_balance_insufficient` | 礼品卡余额不足 |
| 22 | `user_not_found` | modify_user_address 目标用户不存在 |

### Layer 6 — Resource Locks

| # | Block Reason | 触发条件 |
|---|-------------|---------|
| 23 | `duplicate_write_lock` | 同一 lock key 已存在 |
| 24 | `order_already_cancelled_or_locked` | order cancel lock 冲突 |
| 25 | `order_items_already_modified` | order modify_items lock 冲突 |
| 26 | `item_already_returned_or_exchanged` | item lock 冲突 |

### Layer 7 — Idempotency

| # | 机制 | 说明 |
|---|------|------|
| 27 | `idempotency_key` | 基于 session_id + tool_name + arguments + resource_lock 的 stable_hash，去重而非 block |

注：Layer 7 不产生 block reason（它是去重机制），因此实际 block reason 共 **26 个**（#1-#26），加上 idempotency key 生成共 27 个独立行为。

## 文档扩展结构

`docs/portfolio-architecture.md` 第 4 节从当前 ~150 词扩展为 ~1200-1500 词，包含 6 个子节：

### 4.1 设计哲学
- 为什么需要 7 层（每层解决一类独立的安全威胁，层叠防御）
- guard 是最终裁决层，LLM 不能绕过 ToolGateway
- deny-wins 原则如何体现在 guard 的短路逻辑中

### 4.2 架构全景
- ASCII 流程图：用户请求 → ToolGateway.execute() → WriteActionGuard.check() → 7 层依次检查 → pass/block
- 关键不变量：tools never call guard directly
- ToolGateway 是唯一入口点

### 4.3 分层详解
每层独立小节（4.3.0 ~ 4.3.7），格式：
- 检查逻辑简述（1-2 句）
- 产生的 block reason 列表（引用 guard.py 关键代码行）
- 失败时的用户可见行为

### 4.4 覆盖率矩阵
28 行 × 5 列大表：
- Layer | Block Reason | 单元测试方法 | Eval Case ID | 状态

状态标记：
- ✅ 双覆盖（单元测试 + eval case 都有）
- ⚠️ 仅 eval 覆盖（eval case 有但单元测试无）
- ➖ 仅单元测试覆盖（单元测试有但 eval case 无）
- ❌ 无覆盖（防御性代码，被更高层 shadow）

底部汇总行。

### 4.5 审计结论
- 已知弱覆盖区域列表
- 为什么可接受（防御性代码 / 被更高层 catch / 需要极端条件触发）
- 后续 Phase 7 新增写操作时的 guard 扩展检查清单

### 4.6 证据展示
- 截图 1：guard block timeline（Workbench 中 `block_wrong_user_order_access` 截图，标注 block reason、no-write invariant、DB hash 不变）
- 截图 2：successful write timeline（`cancel_pending_order` 截图，标注 confirmation → guard pass → DB mutation → audit log）
- 当前关键指标快照：mutation_error_rate=0, guard_block_rate, no-write cases 全部保持 DB hash 不变
