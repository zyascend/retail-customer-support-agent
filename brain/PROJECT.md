## 一句话定义

一个面向零售客服场景的 LLM tool-calling Agent 作品集项目——展示真实 LLM 工具调用、7 层写安全护栏和全链路可审计运行时三项核心工程能力。

## 解决什么问题

零售客服 Agent 面临三个核心挑战：

1. **写操作安全** — 取消订单、修改支付方式、退换货等操作不能误触发，更不能越权执行。每一次写入都需要用户显式确认，并通过多层护栏检查。
2. **意图消歧** — 用户说"我想改一下订单"，可能意味着取消、换商品、改地址、改支付方式。Agent 必须从非结构化自然语言中提取出结构化的意图和槽位。
3. **全链路可审计** — 每一次决策、每一次工具调用、每一次数据库变更，都必须追溯到具体的策略规则，并提供操作前后的哈希对比和幂等键。

核心原则：**LLM 负责理解意图和选择工具；代码负责工具边界、写入护栏、确认流程和审计证据。**

## 刻意不做什么（非目标）

> 这一节**比"做什么"更重要** —— 防止范围蔓延。
> 每条都要写明**为什么不做**，不然未来读者会以为只是遗漏。

- **不做生产级 SaaS 客服系统**：本项目是 AI Agent 工程作品集，不是面向企业的商业产品。不做 SLA、计费、多租户、真实支付集成。
- **不做通用聊天机器人**：不做闲聊、不做情感陪伴、不做非零售领域的客服。领域限定在 tau-bench retail（电商售后）。
- **不追 SOTA benchmark 排名**：eval 的目标是对比展示"加护栏 vs 不加护栏""LangGraph 编排 vs 单 prompt"的工程收益，不是刷榜。`curated_mvp` 11 个精选 case 覆盖核心链路即可。
- **不做 voice / multi-domain**：只做 text retail，不做 telecom、airline 等多领域，不做语音通道。
- **不做离线降级模式**：项目只保留 `llm` 运行模式。没有 provider 时安全转人工，不提供 offline_demo / deterministic 降级路径。
- **不引入微调模型**：所有 Agent 行为通过 prompt engineering + tool schema design + guard rules 控制，不做模型微调。

## 目标用户 / 使用者

- **AI Agent 工程师**（面试官 / 同行评审）：通过代码、架构文档、eval 报告评估 Agent 工程能力。
- **项目作者本人**：作为作品集展示 + 持续实验 Agent 工程最佳实践的沙盒。
- **不用的人**：不懂代码的 business user、需要开箱即用客服系统的中小企业。

## 项目起源与核心信念

项目起源于 tau-bench（后迁移到 tau2-bench / tau3-bench）retail domain 的 Agent 构建挑战。核心理念：

- **信任但验证**：LLM 可以推理意图，但绝不能让它直接操作数据库。代码护栏是最后一道防线。
- **可审计 > 可工作**：一个能跑但不能解释"为什么这次失败了"的 Agent 在作品集中没有价值。trace、hash、failure taxonomy 是项目的一等公民。
- **单一事实来源**：写操作定义（`action_specs.py`）是唯一的权威注册表，所有其他模块（guard、registry、prompts、retail_adapter）从此派生。新增一种写操作只需改一个文件。
- **范围收紧 > 功能堆叠**：宁可在 11 个精选 case 上做到护栏完整、trace 可回放，也不要铺开到 100+ case 但每项都是半成品。
