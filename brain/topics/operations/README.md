# operations/

> **这里放什么**：运维、流程、打包、部署、runbook。
>
> **判断标准**：问自己 —— "这份文档是在回答：**怎么运维 / 每次要做什么？**"

---

## 典型内容

- **发版流程**：打包命令、发布步骤、RELEASE_CHECKLIST
- **构建 / 打包**：跨平台打包、签名、公证
- **部署日志**：某次部署做了什么 + 遇到的问题
- **运维 runbook**：重置用户数据、管理任务、支持流程
- **变更历史**：CHANGELOG（按版本组织）
- **外部协调**：本仓库怎么与其他仓库 / 团队联动

## 不属于这里

- ❌ 系统设计 / 架构 → `systems/`
- ❌ 未来计划 / 路线图 → `planning/`
- ❌ Bug 追踪 → `feedback/`
- ❌ **PRODUCT_PLAN 属于项目根**，不在 operations/（PRODUCT_PLAN 是产品级"要建什么"；operations 是运维级"怎么做"）

## 文件命名建议

- 大写 + 下划线，例如 `RELEASE_CHECKLIST.md` / `PAYMENT_DEPLOY_LOG.md`
- 运维动词在前，或场景描述清晰，让读者知道何时查
- `CHANGELOG.md` 是惯例名字；保留下划线或不用，跟随项目风格

## 维护纪律

- **RELEASE_CHECKLIST**：每次发版走一遍；缺步骤就补上
- **CHANGELOG**：每次发版更新，不要"以后补"
- **DEPLOY_LOG**：部署的时候记录，写明具体步骤和问题
- 运维文档的"权威性"来自**上次真正走过它的时间** —— 长期没用的流程默认按"假定过期"处理
