# topics/

这个目录装的是项目的**业务 / 技术专题文档** —— 按"**回答什么问题**"分类，不是按业务模块。

## 4 个分类与判断标准

| 子目录 | 装什么 | 判断标准（问自己） |
|---|---|---|
| `systems/` | 系统设计、架构、技术选型 | "这是在回答：怎么设计的？" |
| `operations/` | 运维、流程、打包、部署 | "这是在回答：怎么运维 / 每次发版要做什么？" |
| `planning/` | 路线图、定价策略、计划 | "这是在回答：我们要建什么 / 怎么规划？" |
| `feedback/` | 用户反馈、bug 追踪、triage | "这是在回答：现实是什么 / 用户在告诉我们什么？" |

每个子目录有自己的 README，说明这里放什么、不放什么、命名约定、维护纪律。

## 为什么按"问题维度"而不是"业务模块"

模块会增、会减、会改名、会合并、会拆分。"设计 / 运维 / 计划 / 反馈"这四个问题维度要稳定得多。

一个业务模块（比如"支付"）会散在四个子目录：
- 设计文档在 `systems/`
- 部署日志在 `operations/`
- 定价策略在 `planning/`
- 用户反馈在 `feedback/`

但当你要找某个东西时，你**永远知道该去哪个子目录**。

## vs 接续层（`brain/` 直接子文件）

接续层 5 份文件（PROJECT / MAP / STATUS / DECISIONS / HANDOFF）是**每次会话都要读**的元数据。
专题层（这里）是**按需读** —— `MAP.md` 第 5 节会给每份 topic 文档登记一个"何时读"触发条件。

## 维护纪律

- 每份文档开头写"本文件回答：..."，说清边界
- 新文档必须登记到 `brain/MAP.md` 第 5 节
- 删除 / 废弃的文档必须从 MAP 第 5 节下线
- 不要"最后更新"字段 —— 协议保证最新；真要查，用 `git log -1 --format=%ad <path>`
